## 8. Diagnosed the first real infrastructure bug: bad cwd in Harbor tool execution
Inspection of the trial trace showed the agent was attempting sensible commands, but **every tool execution failed immediately**.

Error pattern:

```text
OCI runtime exec failed: exec failed: unable to start container process:
chdir to cwd ("/workspace") set in config.json failed: no such file or directory
```

Meaning:
- the harness was forcing `cwd="/workspace"`
- Harbor’s task container for this benchmark did not have that directory
- therefore every tool call failed before doing useful work

This was the first true runtime bug exposed by the real Harbor validation.

## 10. Fixed the cwd bug and a second runtime bug
Follow-on implementation then fixed two concrete bugs.

Main fix milestone:
- commit: `ead2e86` — **fix: resolve Harbor tool execution failures**

### Bug A — bad cwd
Actual bug:
- `cwd="/workspace"` was being injected into Harbor tool execution

Fix:
- removed the hardcoded `/workspace`
- allowed Harbor execution to proceed with correct working-directory semantics

### Bug B — `reasoning_content` poisoning later turns
A second issue was also found:
- Gemma/llama.cpp assistant responses included `reasoning_content`
- this field was being preserved in history and re-sent on later turns
- that caused the model server to reject subsequent calls with an error about incompatible assistant prefill / thinking mode

Fix:
- strip `reasoning_content` before appending assistant messages back into conversation history

This second fix was important because without it the agent would crash on later turns even after the cwd issue was fixed.

## Suggested next steps for workshop

### Immediate next steps
1. **Inspect the successful-but-failing trial in detail**
   - What did the agent do?
   - Where did its approach break down?
   - Did it waste turns exploring, fail to plan, or fail to edit productively?

2. **Tighten the system prompt / execution policy**
   - stronger decomposition
   - clearer expectations around inspecting files, then editing, then verifying
   - better stopping / completion behavior

3. **Review tool ergonomics**
   - are file editing tools sufficient?
   - is the model fighting awkward interfaces?
   - does it need better primitives or better examples?

4. **Test on easier or narrower tasks first**
   - before heavy optimizer work, make sure the harness can reliably solve something small but real

### Near-term next steps
5. **Increase run observability**
   - keep logging clear and minimal
   - make trajectory review easy
   - capture why turns were spent the way they were

6. **Only then consider optimizer work**
   - once baseline agent behavior is understandable and at least somewhat competent
   - otherwise the optimizer will just search over a broken or underpowered baseline
